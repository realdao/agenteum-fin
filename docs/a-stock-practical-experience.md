## 1 mootdx：行情层的绝对主力

mootdx 走 TCP 二进制协议，连的是通达信行情服务器群（37 台，端口 7709）。不需要注册、不需要 API Key、不封 IP。

### 它能干什么

- **K 线** ：日/周/月/1分/5分/15分/30分/60分，最多 800 根/次
- **五档盘口** ：实时 46 个字段，含 5 档买卖价量
- **逐笔成交** ：含买卖方向标记（0=买、1=卖、2=中性）
- **季报快照** ：37 个字段——EPS、每股净资产、ROE、净利润、营收、流通股本、总股本…
- **F10 公司资料** ：9 大类文本——最新提示、公司概况、财务分析、股东研究、股本结构…

### K 线的 category 参数（文档不全，实测补齐）

| category 值 | 周期 |
| --- | --- |
| 4 | 日线 |
| 5 | 周线 |
| 6 | 月线 |
| 7 | 1 分钟 |
| 8 | 5 分钟 |
| 9 | 15 分钟 |
| 10 | 30 分钟 |
| 11 | 60 分钟 |

很多教程写 category=9 是日线，这是错的。 **4 才是日线** 。

### 踩过的坑

坑 1：datetime 双重存在  
mootdx 返回的 DataFrame 里， `datetime` 同时出现在 index 和 column 里。直接 `reset_index()` 会报列名重复。正确做法：先 `drop(columns=["datetime"])` ，再 `reset_index()` 。

坑 2：海外 IP 连不上  
TCP 直连通达信服务器，需要国内 IP 才稳定。海外 VPS 会超时。

坑 3：block() 和 minute() 返回空  
这两个接口文档里有，但实测返回空数据。别浪费时间。

坑 4：首次连接测速  
第一次运行会自动测速所有 37 台服务器，生成 `~/.mootdx/config.json` 。之后复用最快的那台。

## 2 腾讯财经：估值数据的最佳来源

腾讯财经的行情接口返回大约 88 个字段，用 `~` 分隔，GBK 编码。不需要注册，不封 IP。

### 全网最大的错误：字段 43 不是 PB

我搜了十几篇腾讯财经 API 的教程，几乎所有人都写"索引 43 = PB（市净率）"。

这是错的。实测校准结果：

| 索引 | 真实含义 | 备注 |
| --- | --- | --- |
| 39 | PE（TTM） |  |
| 43 | 振幅%（不是 PB！） | 全网教程写错 |
| 44 | 总市值（亿） |  |
| 45 | 流通市值（亿） |  |
| 46 | PB（市净率） | 真正的 PB 在这 |
| 47 | 涨停价 |  |
| 48 | 跌停价 |  |
| 52 | PE（静态） |  |

检查你的代码  
如果你用了别人的代码解析腾讯行情，查一下 PB 数据对不对。很多人用了错误的索引 43，算出来的估值全是歪的。

### 腾讯 vs mootdx：互补关系

mootdx 擅长

- K 线（多周期）
- 五档盘口 46 字段
- 逐笔成交（含方向）
- 季报快照 37 字段
- F10 公司资料 9 大类
- PE / PB / 换手率
- 总市值 / 流通市值
- 涨跌停价

腾讯财经擅长

- PE（TTM）/ PE（静态）
- PB（市净率）
- 总市值 / 流通市值
- 换手率
- 涨跌停价
- 季报快照
- F10 公司资料
- 逐笔成交方向

两个源合在一起，你就有了一个不需要 Key、不封 IP 的完整行情 + 估值数据管道。

## 3 akshare：好用但有雷区

akshare 是一个 Python 库，底层爬了一堆网站（东财、新浪、同花顺等）。表面上什么都有，但你得知道哪些接口能用，哪些会炸。

### 我只用 akshare 做这几件事

| 用途 | 接口 | 底层数据源 |
| --- | --- | --- |
| 机构一致预期 EPS | stock\_profit\_forecast\_ths | 同花顺 |
| 个股新闻 | stock\_news\_em | 东财 |
| 财联社快讯 | stock\_info\_global\_cls | 财联社 |
| 全球资讯 | stock\_info\_global\_em | 东财 |
| 三大报表 | stock\_financial\_report\_sina | 新浪 |

为什么行情类接口不能用 akshare  
akshare 底层 **45% 的请求打向东方财富** 。东财有反爬，高频调用会封 IP。行情走 mootdx（TCP 协议，不封 IP），这是测了两周的结论。

坑：东财源的资产负债表解析崩溃  
`stock_balance_sheet_by_report_em` 在 akshare v1.18.54 内部 HTML 解析崩溃，已不能用。替代方案： `stock_financial_report_sina` （新浪源）。

坑：一致预期的 indicator 参数  
`stock_profit_forecast_ths` 的 indicator 参数必须写 `"预测年报每股收益"` 这个完整字符串。写"预测详细指标"有时返回空。机构覆盖 < 3 家时数据不可信，直接输出警告。

## 6 同花顺热点接口：信号层的惊喜

✓入选 — 零鉴权，73ms，当日强势股 + 人工标注题材归因

全网视频和教程里很少有人提到这个接口。零鉴权（只需要 User-Agent），实测 73ms 返回当日约 125 只强势股。

### reason tags：信息密度极高的题材归因

同花顺编辑部 **人工运营标注** 的，不是算法生成的。用 `+` 分隔多个题材标签。

| 股票 | reason tags |
| --- | --- |
| 南威软件 | 算力租赁+Token工厂+AI政务+华为合作 |
| 泰晶科技 | 625MHz晶振+光模块+人形机器人 |
| 众生药业 | 创新药+流感+MASH |

### 5 行代码做题材热度榜

```
from collections import Counter
all_tags = []
for r in df["题材归因"].dropna():
    all_tags.extend([t.strip() for t in str(r).split("+") if t.strip()])
print(Counter(all_tags).most_common(10))
```

题材轮动策略的核心信号  
跑一下这 5 行代码，你就知道今天全市场在炒什么题材。

坑  
盘后才完整，15:30 之后调用才稳定。 `errocode` 字段名是同花顺的拼写错误（少了一个 r），代码里要适配。

## 7 百度股市通 PAE：全网最被低估的 A 股数据源

★本次横评最大发现 — 全网几乎没人介绍过

百度股市通不是百度搜索，它是百度金融的独立产品，有自己的行情数据管道。零鉴权，但需要携带 4 个 Header（缺一不可）。

注意 Accept 头的值是 `application/vnd.finance-web.v1+json` ——不是标准的 `application/json` ，不写这个会返回错误。

### 它能干什么（别人不能干的）

| 能力 | 说明 | 其他源对比 |
| --- | --- | --- |
| 概念板块三维归属 | 一个请求 → 行业（申万一二级）+ 概念（十几个）+ 地域，每个带当日涨跌幅 | akshare 要拼 3 个接口 |
| K 线直接带均线 | 返回时直接带 ma5/ma10/ma20，不需要自己算 | 其他免费源都得自己算 |
| 分钟级资金流向 | 主力/散户/超大单/大单/中单/小单，一天 241 个数据点 | akshare 只到日线级别 |
| 20 日历史资金流 | 含超大单净流入、大单净流入、中单净流入、小单净流入、主力净流入 | 其他免费源无此数据 |

坑 1：ResultCode 返回类型不稳定（最诡异的坑）  
同一个接口，有时返回 int 类型的 `0` ，有时返回 string 类型的 `"0"` 。用 `if d["ResultCode"] == 0` 判断有时会漏掉正确响应。解决：统一用 `str()` 转换后比较。

坑 2：实时资金流的数据格式  
不是 JSON 数组，是分号分隔的字符串： `"time,idx,main,retail,super,large,med,small,price;..."` 。需要自己 split 和 parse。

坑 3：日期参数格式  
用紧凑格式 `20260512` ，不是 `2026-05-12` 。不同接口的日期格式不统一。

坑 4：无官方文档  
百度没有公开这套 API 的文档。字段含义是我对着百度股市通网页一个个对出来的。字段名和返回格式随时可能变。

## 8 iwencai：唯一能做自然语言选股的

!有条件入选 — 需要 API Key + X-Claw Headers

iwencai（同花顺问财）能用自然语言搜索："市值低于100亿的半导体公司，近一年涨幅超过50%"——返回符合条件的股票列表。

2025 年升级 SkillHub 2.0 后，X-Claw Headers 成为强制要求，需要申请 API Key 并携带 6 个特殊 Header。隐藏参数： `size` 默认 10 条，实测可调到 50（文档没写）。

## 9 同花顺北向资金：分钟级外资流向

零鉴权，需要 Host + Referer 两个 Header。返回沪股通 + 深股通的分钟级累计净买入，一天 262 个数据点。

重要区分  
同花顺给的是 **累计净买入** （每分钟从开盘累加），东财推送流给的是 **当前流入流出比** 。两者采样口径不同，建议交叉验证，不是二选一。

东财北向数据断供  
2024 年 8 月之后，东方财富北向资金接口断供——净买额字段返回 NaN 或 0。如果你的量化系统还在依赖东财北向历史数据，赶紧检查一下。

## 10 架构思考：怎么把这些源组合在一起

单个数据源都有短板。关键是按层组合，让每个源只做它最擅长的事。

行情层 mootdx（TCP）+ 腾讯财经（HTTP）  
→ K线 + 盘口 + 估值，零鉴权不封IP  
  
研报层 东财研报 + akshare + iwencai  
→ 研报列表 + PDF + 一致预期 + NL搜索  
  
信号层 同花顺 + 百度PAE + akshare + 东财DC  
→ 强势股 + 题材归因 + 北向资金 + 概念板块 + 资金流 + 龙虎榜  
  
新闻层 akshare × 3  
→ 个股新闻 + 财联社 + 全球资讯  
  
基础数据 mootdx finance + F10  
→ 季报快照 37 字段 + 公司资料 9 大类  
  
公告层 巨潮 cninfo  
→ 沪深北全量公告

### 为什么行情和估值分两个源？

mootdx 走 TCP 协议，给你交易层数据（价格、盘口、成交）。但它不计算 PE/PB——这些是衍生指标，需要价格 + 财务 + 股本数据才能算。腾讯财经已经帮你算好了。两个源加起来 = 完整的行情 + 估值管道，都不封 IP，都不需要 Key。

### 为什么 akshare 只做低频？

akshare 底层 45% 打向东财，东财有反爬。把 akshare 限制在低频场景：研报（一天查几次）、新闻（一天拉几次）、一致预期 EPS（一天一次）。行情走 mootdx，资金流走百度，绝对不碰 akshare 的东财源行情接口。

## 11 估值框架：30x PE 锚点

A 股成长股的估值，我统一用 30 倍 PE 做锚点。

```
壁垒 → 增速 → PE消化 → PEG校验

1. 有壁垒吗？没有 → 排除
2. CAGR > 30% 吗？没有 → 不值得给成长股溢价
3. PE 消化到 30x 要多久？
   < 2 年 → 合理估值
   2-4 年 → 边缘区
   > 4 年 → 太贵
4. PEG 校验：
   < 1   → 便宜
   1-1.5 → 合理
   > 1.5 → 贵
```

公式：

| 指标 | 公式 |
| --- | --- |
| Forward PE | 当前股价 / 一致预期 EPS |
| PEG | Forward PE / (EPS CAGR × 100) |
| PE 消化时间 | ln(当前PE / 30) / ln(1 + CAGR) |

一致预期 EPS 从 akshare 同花顺源拿，当前股价从腾讯财经拿， **30 秒出一个完整估值结论** 。
